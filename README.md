## 🎯 Мета

Розширити backend-застосунок Library Management System:

- реалізувати **скидання пароля** через email (запит на скидання → лист з токеном → зміна пароля)
- додати можливість **завантаження та оновлення аватарки** користувача (file upload)
- навчитися працювати з **відправкою email** (nodemailer / SendGrid)
- навчитися обробляти **завантаження файлів** (multipart/form-data)
- зберегти всю попередню функціональність з ДЗ-2

---

## 📝 Опис завдання

Ви розширюєте проєкт з ДЗ-2:

1. **Скидання пароля** — користувач може запросити скидання пароля, вказавши email. На пошту надходить лист з посиланням (або кодом). За цим посиланням можна встановити новий пароль.
2. **Аватарка користувача** — авторизований користувач може завантажити зображення як свою аватарку. Аватарка зберігається на сервері (локальна файлова система) та повертається як URL у профілі користувача.

❗ Усі ендпоїнти з ДЗ-2 (книги, позики, аутентифікація, авторизація) мають продовжувати працювати.

### 🔐 Аутентифікація (існуючі + нові ендпоїнти)

| Метод | Endpoint                       | Опис                                 | Доступ    |
| ----- | ------------------------------ | ------------------------------------ | --------- |

| POST  | `/auth/reset-password`         | **Нове.** Встановлення нового пароля | Публічний |

---

#### `POST /auth/request-password-reset`

**Тіло запиту:**

```json
{
  "email": "user@example.com"
}
```

**Поведінка:**

- Якщо користувач з таким email існує — згенерувати токен скидання та надіслати лист на пошту.
- Якщо email не знайдено — **все одно повернути успішну відповідь** (щоб не розкривати, які email зареєстровані).
- Токен має бути обмежений у часі (наприклад, 10–15 хвилин).

**Відповідь (завжди):**

```json
{
  "message": "Якщо вказаний email зареєстрований, лист з інструкціями надіслано."
}
```

---

#### `POST /auth/reset-password`

**Тіло запиту:**

```json
{
  "token": "eyJhbGciOiJIUzI1NiIs...",
  "password": "newSecurePassword123"
}
```

**Поведінка:**

- Перевірити валідність токена (не протермінований, коректний підпис).
- Захешувати новий пароль та оновити запис користувача в БД.
- При невалідному або протермінованому токені — повернути **400 Bad Request**.

**Відповідь (успіх):**

```json
{
  "message": "Пароль успішно змінено."
}
```

### 🖼 Аватарка користувача (нові ендпоїнти)

| Метод  | Endpoint           | Опис                            | Доступ               |
| ------ | ------------------ | ------------------------------- | -------------------- |
| POST   | `/users/me/avatar` | Завантажити / замінити аватарку | Тільки авторизований |
| DELETE | `/users/me/avatar` | Видалити аватарку               | Тільки авторизований |

---

#### `POST /users/me/avatar`

**Тіло запиту:** `multipart/form-data` з полем `avatar` (файл зображення).

**Поведінка:**

- Прийняти файл зображення (JPEG, PNG).
- Валідувати тип файлу та розмір (максимум 5 МБ).
- Зберегти файл на сервері (наприклад, у папку `uploads/avatars/`).
- Оновити поле `avatarUrl` у профілі користувача (зберегти шлях або URL до файлу).
- Якщо у користувача вже є аватарка — **видалити старий файл** перед збереженням нового.

**Відповідь:**

```json
{
  "message": "Аватарку успішно оновлено.",
  "avatarUrl": "/uploads/avatars/uuid-filename.jpg"
}
```

---

#### `DELETE /users/me/avatar`

**Поведінка:**

- Видалити файл аватарки з файлової системи.
- Встановити `avatarUrl = null` у профілі користувача.
- Якщо аватарки немає — повернути **404 Not Found**.

**Відповідь:**

```json
{
  "message": "Аватарку видалено."
}
```

### 👤 Users (оновлені відповіді)

Ендпоїнти `GET /users/me`, `GET /users`, `GET /users/:id` тепер повертають поле `avatarUrl` у відповіді:

```json
  "email": "user@example.com",

  "role": "USER",

```

Якщо аватарка не завантажена — `avatarUrl: null`.

## 📑 Валідація

- **`/auth/request-password-reset`:** поле `email` — обовʼязкове, валідний формат email.
- **`/auth/reset-password`:** поле `token` — обовʼязкове, непорожній рядок; поле `password` — мінімум 8 символів.
- **`POST /users/me/avatar`:** перевірка MIME-типу файлу (`image/jpeg`, `image/png`), перевірка розміру (≤ 5 МБ). При невалідному файлі — **400 Bad Request** з описом помилки.

## 📌 Бізнес-логіка

### Скидання пароля

Реалізувати потрібно **один із двох підходів** (на вибір студента):

#### Підхід A: Stateless (JWT-токен)

- При запиті скидання — підписати JWT з payload `{ email }` та коротким `expiresIn` (наприклад, `"10m"`).
- Надіслати токен у листі (як частину посилання або окремим кодом).
- При скиданні — верифікувати JWT, витягти email, оновити пароль.
- **Плюс:** не потрібні додаткові поля в БД.
- **Мінус:** токен неможливо відкликати до закінчення терміну дії.

#### Підхід B: Stateful (токен у БД)

- При запиті скидання — згенерувати випадковий токен (наприклад, `crypto.randomUUID()` або `crypto.randomBytes(32).toString("hex")`), зберегти його хеш та дату закінчення в БД (додаткові поля `resetToken`, `resetTokenExpires` у моделі User, або окрема таблиця).
- Надіслати оригінальний токен у листі.
- При скиданні — знайти користувача за токеном, перевірити термін дії, оновити пароль, видалити/інвалідувати токен.
- **Плюс:** токен можна відкликати (видалити з БД).
- **Мінус:** потрібно зберігати стан у БД.

### Відправка email

- Використовувати **nodemailer** з SMTP-транспортом (наприклад, Brevo, Mailtrap, Gmail) або **SendGrid**.
- Лист має містити посилання або код для скидання пароля.
- SMTP-налаштування зберігати в **змінних оточення** (`.env`), не в коді.

### Завантаження аватарок

- Використовувати **multer** як Express middleware для обробки `multipart/form-data`.
- Файли зберігати у папку на сервері (наприклад, `uploads/avatars/`).
- Імʼя файлу має бути **унікальним** (наприклад, використовувати UUID або userId у назві).
- Роздавати завантажені файли через `express.static`.
- При завантаженні нової аватарки — видаляти попередній файл з файлової системи.
- При видаленні користувача (якщо є такий ендпоїнт) — також видаляти файл аватарки.
