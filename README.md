## Library Management System (База даних + Auth)

**Курс:** Backend розробка на Node.js  
**Дедлайн:** кінець 9-го тижня  
**Максимум:** 20 балів  
**Бонус:** 5 балів за refresh tokens або OAuth (Google/GitHub)

**Передумова:** виконане ДЗ-1 (Library Management System in-memory) або еквівалентний функціонал.

---

## 🎯 Мета

Розширити backend-застосунок Library Management System:

- **перенести зберігання даних** з оперативної памʼяті в **базу даних**
- реалізувати **реєстрацію та вхід** (аутентифікація)
- захистити API за допомогою **авторизації** (хто має право виконувати дії)
- зберегти REST API та бізнес-логіку з ДЗ-1

---

## 📝 Опис завдання

Ви розширюєте проєкт з ДЗ-1:

1. **База даних** — книги, користувачі та позики зберігаються в БД (SQLite або PostgreSQL). Після перезапуску дані зберігаються.
2. **Аутентифікація** — користувач може зареєструватися та увійти; після входу отримує токен (наприклад, JWT).
3. **Авторизація** — частина ендпоінтів доступна лише авторизованим користувачам; деякі дії (наприклад, видалення книги) — лише адміну.

❗ Ендпоїнти для книг, позик тощо мають працювати як у ДЗ-1, але з читанням/записом у БД та перевіркою прав доступу де потрібно.

- **Node.js**
- **Express**

## 🌐 REST API

### 🔐 Аутентифікація (публічні)

| Метод | Endpoint | Опис |
| ----- | -------- | ---- |
| POST | `/auth/register` | Реєстрація (email, пароль, name) |
| POST | `/auth/login`    | Вхід (email, пароль) → повертає JWT |

**Приклад тіла для реєстрації/логіну:**

```json
{
  "email": "user@example.com",
  "password": "securePassword123",
  "name": "Іван Петренко"
}
```

(Поле `name` обовʼязкове тільки для `/auth/register`.)

**Відповідь при успішному логіні:**

```json
{
  "token": "eyJhbGciOiJIUzI1NiIs...",
  "user": {
    "id": "uuid",
    "email": "user@example.com",
    "name": "Іван Петренко",
    "role": "USER"
  }
}
```

## 🔑 Авторизація через JWT

- Клієнт після логіну зберігає токен і надсилає його в заголовку:
  - `Authorization: Bearer <token>`
- Сервер перевіряє токен на захищених маршрутах; при невалідному/відсутньому токені повертає **401 Unauthorized**.
- За потреби перевіряти роль (наприклад, тільки ADMIN) — при відсутності прав повертати **403 Forbidden**.

---

## 📑 Валідація

- Як у ДЗ-1: усі `POST` та `PUT` валідовувати через **Zod**.
- Для `/auth/register` та `/auth/login`: валідувати `email`, `password`; для реєстрації — також `name`.
- Пароль: мінімальна довжина (наприклад, 8 символів); при помилці — **400** з описом.

---

## 📌 Бізнес-логіка

### Реєстрація та логін

- **Реєстрація:** email має бути унікальним; пароль зберігати лише у вигляді хешу.
- **Логін:** перевірити email та пароль (порівняння з хешем); при успіху видати JWT (в payload можна зберігати `userId`, `email`, `role`).

### Книги та позики

- Логіка видачі/повернення книг — як у ДЗ-1 (книга недоступна → не видавати; при поверненні — `available = true`, позика `RETURNED`).
- Всі операції з книгами та позиками виконуються через БД (створення, оновлення, читання).
